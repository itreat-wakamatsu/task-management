## リレーションシップ説明

### auth.users → app_tasks
- **多重度:** 1対多（1ユーザーは複数タスクを持つ）
- **結合列:** `app_tasks.user_id = auth.users.id`
- **RLS:** `auth.uid() = user_id` で本人のみアクセス可

### auth.users → app_records
- **多重度:** 1対多（1ユーザーは複数日次記録を持つ）
- **ユニーク制約:** `(user_id, target_date)` でユーザー×日付が重複しない

### auth.users → backlog_tokens
- **多重度:** 1対1（1ユーザーにつき1トークンのみ）
- **主キー:** `user_id`（1ユーザーに1行のみ）

### clients → projects
- **多重度:** 1対多（1クライアントは複数案件を持つ）
- **結合列:** `projects.client_id = clients.id`

### projects → project_categories
- **多重度:** 1対多（1案件は複数カテゴリを持つ）
- **自己参照:** `project_categories.parent_id` で2階層を表現

### app_tasks → clients / projects / project_categories
- **多重度:** 多対1（複数タスクが同じクライアント/案件/カテゴリを参照）
- `category_id` は第1階層カテゴリ
- `subcategory_id` は第2階層カテゴリ（`parent_id IS NOT NULL` の行）

### app_records → app_record_details
- **多重度:** 1対多（1日次記録は複数の明細を持つ）
- **結合列:** `app_record_details.record_id = app_records.id`

### app_tasks → app_record_details
- **多重度:** 1対多（1タスクは複数の記録明細に紐付けられる）
- **nullable:** `task_id IS NULL` の場合はタスク未紐付け

## 論理削除の扱い

以下のテーブルは物理削除せず `deleted_at` で論理削除:

| テーブル | 削除フラグ列 | フィルター条件 |
|---------|------------|-------------|
| clients | deleted_at | `IS NULL` |
| projects | deleted_at | `IS NULL` |
| project_categories | deleted_at | `IS NULL` |
| app_tasks | deleted_at | `IS NULL` |

`app_records`, `app_record_details`, `backlog_tokens` は論理削除なし（物理削除）。
