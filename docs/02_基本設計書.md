## 1. システム概要

### 1-1. システム名

**タスクタイマー** (Task Timer)

### 1-2. システム概要

タスクタイマーは、Googleカレンダーと連携してタスクの作業時間を計測・記録・分析するWebアプリケーションです。予定時間と実績時間を比較することで、時間管理の習慣化を支援します。

### 1-3. 対象ユーザー

- フリーランス・個人事業主（時間管理・請求のため）
- 社内SE・システムエンジニア（案件・タスクの工数管理）
- プロジェクトマネージャー（進捗・効率の把握）

### 1-4. システム稼働環境

| 項目 | 内容 |
|------|------|
| 提供形態 | Webアプリケーション（SPA） |
| アクセス方法 | Webブラウザ（Chrome推奨） |
| ホスティング | Vercel |
| データベース | Supabase（PostgreSQL） |

## 2. システム目的・要件

### 2-1. 解決する課題

1. **時間の過小見積もり:** 予定と実績の差を可視化し、見積もり精度を向上させる
2. **工数記録の煩雑さ:** カレンダーと自動連携し、最小限の操作で記録できる
3. **案件別コスト把握:** クライアント・案件・カテゴリ別に集計し、収支分析を可能にする

### 2-2. 機能要件

| ID | 要件 | 優先度 |
|----|------|--------|
| FR-01 | Googleカレンダーの予定をタイマー対象として表示する | 必須 |
| FR-02 | タスクごとに開始・一時停止・終了を記録する | 必須 |
| FR-03 | 計画時間に対する実績時間を計算・表示する | 必須 |
| FR-04 | タスクマスターを管理（CRUD）できる | 必須 |
| FR-05 | 実績データをクライアント・案件・カテゴリ別に集計する | 必須 |
| FR-06 | カレンダー予定とタスクを手動・自動で紐付ける | 必須 |
| FR-07 | Backlog課題をタスクとしてインポートする | 推奨 |
| FR-08 | タスクに開始日・期日を設定できる | 推奨 |
| FR-09 | 繰り返しタスクをフラグ管理できる | 推奨 |

### 2-3. 非機能要件

| ID | 要件 | 目標値 |
|----|------|--------|
| NF-01 | 応答速度 | 初期ロード3秒以内、操作応答1秒以内 |
| NF-02 | 可用性 | 99%以上（Vercel・Supabase SLA依存） |
| NF-03 | セキュリティ | RLSによるデータ分離、HTTPS通信 |
| NF-04 | ブラウザ対応 | Chrome最新版、Safari最新版 |
| NF-05 | デバイス対応 | デスクトップ（最小幅 860px）推奨 |

## 5. 外部インターフェース

### 5-1. Google Calendar API

| 項目 | 内容 |
|------|------|
| 認証方式 | OAuth 2.0（Authorization Code Flow） |
| スコープ | `https://www.googleapis.com/auth/calendar` |
| 使用エンドポイント | GET /calendars/primary/events |
| 用途 | 当日の予定一覧取得 |
| リダイレクトURI | `https://{supabase-project}.supabase.co/auth/v1/callback` |

### 5-2. Backlog API

| 項目 | 内容 |
|------|------|
| 認証方式 | OAuth 2.0（Authorization Code Flow） |
| 使用エンドポイント | GET /api/v2/users/myself, GET /api/v2/issues |
| 用途 | 担当課題の取得・ユーザー情報取得 |
| トークン中継 | `/api/backlog-token` Vercel Serverless Function経由 |

### 5-3. Supabase

| 項目 | 内容 |
|------|------|
| 認証 | Supabase Auth（Google OAuth連携） |
| データベース | PostgreSQL（PostgRESTでREST API） |
| SDK | `@supabase/supabase-js` v2 |
| セキュリティ | Row Level Security（RLS）有効 |

## 7. 認証・認可設計

### 7-1. 認証フロー

1. ユーザーがGoogleログインを選択
2. SupabaseがGoogle OAuth 2.0フローを開始
3. GoogleがSupabaseのコールバックURLにリダイレクト
4. SupabaseがJWTセッションを発行
5. フロントエンドがJWTをLocalStorageに保存
6. 以降のAPIリクエストにJWTを付与

### 7-2. Row Level Security (RLS)

すべてのユーザー固有テーブルにRLSを設定:

```sql
-- app_tasks の例
CREATE POLICY "Users see own tasks"
  ON app_tasks FOR ALL
  USING (auth.uid() = user_id);
```

### 7-3. APIキー管理

| キー | 保存場所 | 公開範囲 |
|------|---------|---------|
| Supabase Anon Key | フロントエンド環境変数 | 公開（RLSで保護） |
| Google Client ID | フロントエンド環境変数 | 公開 |
| Backlog Client ID | フロントエンド環境変数 | 公開 |
| Backlog Client Secret | Vercel環境変数（サーバーのみ） | 非公開 |

## 8. 非機能要件

### 8-1. パフォーマンス

- 初期ロード: 3秒以内（Vercel CDN利用）
- API応答: 1秒以内（React Query 5分キャッシュ）
- タイマー更新: 1秒間隔（`setInterval`）

### 8-2. セキュリティ

- HTTPS通信必須
- RLSによるユーザーデータ分離
- Backlog Client SecretはサーバーサイドAPIのみで使用
- OAuth stateパラメータによるCSRF対策

### 8-3. 可用性

- Vercel Free Tierの稼働率に準拠
- Supabase Free Tierの稼働率に準拠
- SPAのためオフライン時は機能しない

### 8-4. 保守性

- コンポーネント単位のCSS Modules
- Zustandによる一元状態管理
- Supabase Migrationsによるスキーマ管理
