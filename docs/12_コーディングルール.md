## 1. 全般

### 1-1. コードスタイル

- インデント: **スペース2つ**
- 文字コード: **UTF-8**
- 改行コード: **LF**（`.gitattributes` で統一）
- セミコロン: **なし**（Vite/ES Modules スタイル）
- クォート: **シングルクォート** `'` （JSX属性のみダブルクォート）

```javascript
// ✓ 良い例
import { useState } from 'react'
const greeting = 'Hello'

// ✗ 悪い例
import { useState } from "react";
const greeting = "Hello";
```

### 1-2. コメント

- 自明なコードにはコメントをつけない
- 複雑なロジックや非自明な判断にのみコメントをつける
- 日本語コメントを使用して良い（チームは日本語話者）

```javascript
// ✓ 良い例（非自明なロジックに説明）
// 手動調整値を優先し、なければ自動計算を使う
const elapsed = event.overrideElapsedMs ?? calcNetElapsed(event, isPaused, pausedAt)

// ✗ 悪い例（自明なことをコメント）
// ステートを更新する
setLoading(true)
```

### 1-3. console.log

- 開発中のデバッグ用 `console.log` はコミット前に削除する
- エラーログは `console.error` を使用し、残しても良い

## 2. JavaScript / React

### 2-1. コンポーネント定義

- **関数コンポーネント**のみ使用（クラスコンポーネント禁止）
- デフォルトエクスポートを使用

```javascript
// ✓ 良い例
export default function MyComponent({ title, onClose }) {
  return <div>{title}</div>
}

// ✗ 悪い例（クラスコンポーネント）
export default class MyComponent extends React.Component { ... }
```

### 2-2. Props

- 分割代入でpropsを受け取る
- propsの型コメントは不要（TypeScript未使用）

```javascript
// ✓ 良い例
export default function TaskCard({ event, isActive, onStart }) { ... }

// ✗ 悪い例
export default function TaskCard(props) {
  const { event, isActive, onStart } = props
  ...
}
```

### 2-3. Hooks のルール

- **Hooksは常にコンポーネントのトップレベルで呼ぶ**
- 条件分岐・ループ・早期returnの**前**に呼ぶ

```javascript
// ✓ 良い例
export default function TimerHero({ event }) {
  const client  = useStore(s => s.clients.find(c => c.id === event?.task?.client_id))
  const project = useStore(s => s.projects.find(p => p.id === event?.task?.project_id))

  if (!event) return <div>待機中</div>  // 早期return は Hooks の後

  return <div>...</div>
}

// ✗ 悪い例（Rules of Hooks 違反）
export default function TimerHero({ event }) {
  if (!event) return <div>待機中</div>  // ← Hooks の前に return

  const client = useStore(s => s.clients.find(...))  // 違反！
}
```

### 2-4. 非同期処理

- `async/await` を使用（`.then()/.catch()` チェーンは避ける）
- Supabaseの `{ data, error }` は必ずエラーチェックをする

```javascript
// ✓ 良い例
const { data, error } = await supabase.from('app_tasks').select('*')
if (error) {
  console.error('Failed to fetch tasks:', error)
  return
}

// ✗ 悪い例
const { data } = await supabase.from('app_tasks').select('*')
// errorを無視している
```

### 2-5. 条件レンダリング

```javascript
// ✓ 良い例（&&で短い場合）
{client && <span>{client.name}</span>}

// ✓ 良い例（三項演算子）
{isPaused ? '再開' : '一時停止'}

// ✗ 悪い例（複雑な&& でfalsy値が表示される）
{count && <span>{count}</span>}  // count=0 のとき "0" が表示される
// → {count > 0 && <span>{count}</span>} と書く
```

### 2-6. イベントハンドラ命名

- `handle` プレフィックスをつける

```javascript
// ✓ 良い例
function handleStart(eventId) { ... }
function handleKeyDown(e) { ... }

// ✗ 悪い例
function startTimer(eventId) { ... }  // コンポーネント内のイベントハンドラなのに「handle」なし
```

### 2-7. useEffect

- クリーンアップが必要な副作用は必ずreturnで後処理をする
- 依存配列を明示する

```javascript
// ✓ 良い例
useEffect(() => {
  if (!activeEventId) return
  const id = setInterval(() => setTick(t => t + 1), 1000)
  return () => clearInterval(id)
}, [activeEventId])
```

### 2-8. 変数宣言

- `const` を優先、変更が必要な場合のみ `let`
- `var` は使用禁止

### 2-9. Optional Chaining

- nullが入り得るオブジェクトへのアクセスには `?.` を使用

```javascript
// ✓ 良い例
const clientId = event?.task?.client_id

// ✗ 悪い例
const clientId = event && event.task && event.task.client_id
```

## 3. CSS Modules

### 3-1. ファイル命名

- コンポーネントと同名: `ComponentName.module.css`
- 同じディレクトリに配置

### 3-2. クラス命名

- **camelCase** を使用
- BEMは不要（Modulesで名前衝突しないため）

```css
/* ✓ 良い例 */
.cardHeader { ... }
.clientRow { ... }
.progressFill { ... }

/* ✗ 悪い例（BEM風） */
.card__header { ... }
.card--active { ... }
```

### 3-3. 状態クラス

- 状態は短い文字列で表現し、動的にJSから適用

```javascript
// コンポーネント側
<div className={`${styles.box} ${styles[state]}`}>

// CSS側
.box.run  { border-color: #3b82f6; }
.box.pause { border-color: #f59e0b; }
.box.over  { border-color: #ef4444; }
```

### 3-4. CSS変数の使用

- テーマカラーには必ずCSS変数を使用
- ハードコードが許可されるのは: 外部ブランドカラー（Backlog緑、DEVオレンジ、Google色）のみ

```css
/* ✓ 良い例 */
color: var(--color-text-primary);
background: var(--color-bg-secondary);
border: 0.5px solid var(--color-border);

/* ✓ 例外（ブランドカラー） */
color: #f59e0b;  /* DEVモードのオレンジ */

/* ✗ 悪い例 */
color: #1a1a1a;  /* CSS変数を使うべき */
```

### 3-5. 単位

- フォントサイズ: `rem`
- パディング・マージン: `rem` または `px`
- ボーダー幅: `0.5px` または `1px`
- ブレークポイント: `px`

## 4. Supabase クエリ

### 4-1. 基本パターン

```javascript
// ✓ 正しいパターン
const { data, error } = await supabase
  .from('app_tasks')
  .select('*')
  .eq('user_id', userId)
  .is('deleted_at', null)
  .order('usage_count', { ascending: false })

if (error) throw error
```

### 4-2. FK Joinは使わない

**現在のDBにFKが設定されていないため、FK Joinは使用禁止。**
2ステップ取得パターンを使用すること。

```javascript
// ✗ 禁止（FKがないためPGRST100エラー）
const { data } = await supabase
  .from('app_record_details')
  .select('*, app_records!inner(user_id)')

// ✓ 正しいパターン（2ステップ）
const { data: records } = await supabase
  .from('app_records')
  .select('id, target_date')
  .eq('user_id', userId)

const ids = records.map(r => r.id)
const { data: details } = await supabase
  .from('app_record_details')
  .select('*')
  .in('record_id', ids)
```

### 4-3. RLSを信頼する

`user_id = userId` の条件は必須だが、セキュリティはRLSに依存する。
フロント側でも条件を付けるのは二重確認のため（良い習慣）。

### 4-4. upsert

Backlogトークンなど1ユーザー1行のテーブルはupsertを使用:

```javascript
await supabase
  .from('backlog_tokens')
  .upsert({ user_id: userId, ...data })
```

## 5. 状態管理（Zustand）

### 5-1. セレクター

- コンポーネントで必要な値だけをセレクターで取得する（不要な再レンダリング防止）

```javascript
// ✓ 良い例（必要な値だけ取得）
const activeEventId = useStore(s => s.activeEventId)
const client = useStore(s => s.clients.find(c => c.id === clientId))

// ✗ 悪い例（ストア全体を取得）
const store = useStore()
```

### 5-2. ストアへの直接操作禁止

- コンポーネントからDBを直接操作した後は必ずストアも更新する

```javascript
// ✓ 正しいパターン
await supabase.from('app_tasks').update({ ... }).eq('id', id)
setAppTasks(appTasks.map(t => t.id === id ? { ...t, ...patch } : t))  // ストアも更新
```

### 5-3. 非同期アクションはストア内に定義

- `supabase.from(...)` の呼び出しはストアのアクション関数またはコンポーネントのuseEffect内に書く
- コンポーネントのイベントハンドラには最小限の処理のみ

## 6. ファイル・ディレクトリ命名

| 種別 | 規則 | 例 |
|------|------|-----|
| Reactコンポーネント | PascalCase | `TaskCard.jsx` |
| CSSモジュール | PascalCase + `.module.css` | `TaskCard.module.css` |
| Hooksファイル | camelCase（useから始める） | `useTimer.js` |
| ユーティリティ | camelCase | `autoLink.js`, `googleCalendar.js` |
| ディレクトリ | PascalCase（コンポーネント）、camelCase（その他） | `TaskManager/`, `lib/`, `hooks/` |
| Vercel API | camelCase + ハイフン | `backlog-token.js` |
| SQLマイグレーション | 連番 + スネークケース | `001_init.sql` |

## 7. コミット・ブランチ

### 7-1. コミットメッセージ（Conventional Commits）

```
<type>: <内容>

type:
  feat     - 新機能
  fix      - バグ修正
  refactor - リファクタリング（機能変更なし）
  style    - コードスタイルのみの変更
  docs     - ドキュメントのみの変更
  chore    - ビルド・CI・依存関係の変更
  test     - テストの追加・修正
```

**例:**
```
feat: Backlog OAuth連携とタスクインポート機能を追加
fix: TimerHeroのReact Hooks順序違反を修正
docs: テーブル定義書を追加
```

### 7-2. ブランチ命名

```
feature/<機能名>   - 新機能（例: feature/analytics-monthly）
fix/<バグ名>       - バグ修正（例: fix/timer-pause-log）
docs/<内容>        - ドキュメント（例: docs/coding-rules）
chore/<内容>       - メンテナンス（例: chore/update-deps）
```

### 7-3. マージフロー

```
feature/* → dev → main
```

- `main` への直接コミット禁止
- `main` はリリース済みの安定バージョンのみ

## 8. 禁止事項

| 禁止 | 理由 | 代替 |
|------|------|------|
| `var` の使用 | スコープ問題 | `const` / `let` |
| クラスコンポーネント | 統一性 | 関数コンポーネント + Hooks |
| `dangerouslySetInnerHTML` | XSSリスク | 通常のJSXレンダリング |
| `import.meta.env.VITE_BACKLOG_CLIENT_SECRET` | Client Secretは非公開 | `/api/backlog-token` 経由 |
| Supabase FK Join（`!inner`） | DBにFKがないためPGRST100 | 2ステップ取得 |
| `console.log` のコミット | デバッグ残留 | コミット前に削除 |
| `any` 型（将来TypeScript導入時） | 型安全性の無効化 | 適切な型定義 |
| テーマカラーのハードコード | ダークモード対応不可 | CSS変数 |
| 直接的なDOM操作（`document.querySelector`等） | Reactのレンダリングと競合 | `ref` を使用 |
